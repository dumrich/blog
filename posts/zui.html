Building a high level TUI framework in Rust

Source: xkcd: Command Line Fu

If you’re reading this blog post, you love Unix. And if you love Unix, you love the command line. You may not know it yet, but I assure you that it will eventually ring true.

A step above the command line is the TUI (Terminal User Interface). A TUI, as you may know, is a more interactive command line application that allows for user input, and cool looking widgets. Some of the more popular of these applications are vim , htop, or lynx.

In the spirit of recreating things that I love, I decided to build a TUI framework in Rust.

Repo: https://git.dumrich.com/zui/

Crate(s): https://crates.io/crates/zui-core, https://crates.io/crates/zui-widgets

Make sure you import the two crates above!

Escape Codes

The most essential concept to creating terminal applications are escape codes or escape sequences. They’re just strings that the terminal interprets to change the way it displays something. Check here for a list of the most important ones.

Escape codes are the most direct way to control the terminal, including things like:

  • Colors

  • Styles

  • Cursor Movement

  • Terminal Clearing

  • User Input

  • and more…

First up, is Colors and Styles

Colors and Styles

These two are probably the easiest to implement and use as a style and a color map directly to an ANSI escape code:

pub enum Color {
          Black,
          Red,
          Green,
          Yellow,
          Blue,
          Purple,
          Cyan,
          White,
          ...
      }

I first created an Enum as an abstraction over a raw code.

match color {
              Color::Black => "\u{001b}[30m",
              Color::Red => "\u{001b}[31m",
              Color::Green => "\u{001b}[32m",
              Color::Yellow => "\u{001b}[33m",
              Color::Blue => "\u{001b}[34m",
              Color::Purple => "\u{001b}[35m",
              Color::Cyan => "\u{001b}[36m",
              Color::White => "\u{001b}[37m",
              ...
      }

I then match each enum variant to a &'static str escape code.

Styles are largely the same:

pub enum Style {
          Dim,
          Reverse,
          Invisible,
          Bold,
          Italic,
          Underline,
          Blinking,
          Strike,
          ...
      }
match style {
              Dim => "\u{001b}[2m",
              Reverse => "\u{001b}[7m",
              Invisible => "\u{001b}[8m",
              Bold => "\u{001b}[1m",
              Italic => "\u{001b}[3m",
              Underline => "\u{001b}[4m",
              Blinking => "\u{001b}[5m",
              Strike => "\u{001b}[9m",
              ...
          }

We could use these functions as so:

use zui_core::style::{set, Style};
      use zui_core::color::{set, fg};
      
      
      fn main() {
          println!("{}I'm Red{} Now I'm bold", fg(color::Color), set(Style::Bold));
      }

Cursor and Clearing

Cursor manipulation and clearing are pretty similar to styles and colors.

Upon writing a cursor escape code to the stdout, the terminal will move the cursor to a specific coordinate (like x, y). Same for one of the clear codes.

You can see a full list of all methods here.

Here is an example of the codes required to set the cursor position and change the type of cursor:

    pub fn set_cursor_to(&mut self, x_pos: u16, y_pos: u16) -> io::Result<()> {
              if x_pos <= self.rel_size.0 && y_pos <= self.rel_size.1 {
                  let result = write!(self.stdout, "\u{001b}[{};{}f", y_pos, x_pos)?;
                  self.stdout.flush().unwrap();
                  self.x_pos = x_pos;
                  self.y_pos = y_pos;
                  Ok(result)
              } else {
                  panic!("Cursor set to out of bounds");
              }
          }

The code “\u{001b}[{x};{y}f”. This specific code expects the coordinates of the cursor location.

Escape codes can also let us change the type of the cursor:

    pub fn blinking_block(&mut self) -> io::Result<()> {
              self.cursor_mode = Cursor::BlinkingBlock;
              let result = write!(self.stdout, "\u{001b}[1 q")?;
              self.stdout.flush().unwrap();
              Ok(result)
          }

Now let’s see how we would use some of these clearing and cursor methods:

use zui_core::term;
      use std::io;
      
      fn main() {
          let mut stdout = io::stdout();
          let mut terminal = term::Terminal(&mut stdout).unwrap();
      
          // Clear the current Line
          terminal.clear_line().unwrap(); 
      
          // Move cursor
          terminal.set_cursor_to(1, 4).unwrap();
      }

Input and Raw Mode

This is where it starts to get complicated. Let me explain.

When you’re running your python program using input(), or a basic program that collects some sort of input, your terminal is in what is known as Cannonical Mode. Cannonical mode is good enough for simple programs as it will write out any character the user writes to stdout and only processes the text when the user hits enter.

While this is great for most programs, we want our terminal framework to go above and beyond. We want to capture every user input and do some extra processing. For example, htop doesn’t display every input character by default. This is because raw mode, the terminal mode we want, achieves a few things:

  • Doesn’t display all characters

  • Be able to process all characters as soon as they’re pressed (opposed to pressing enter to get the bytes)

There was no way to do this easily in Rust, so I needed to use libc.

// Enter Raw mode
      pub use libc::termios as Termios;
      
      extern "C" {
          fn tcgetattr(fd: raw::c_int, termios: &mut Termios) -> raw::c_int;
      
          fn tcsetattr(fd: raw::c_int, optional_actions: raw::c_int, termios: &mut Termios)
              -> raw::c_int;
      
          fn cfmakeraw(termios: &mut Termios);
      
      }

These are the functions to enter raw mode. We are able to use libc functions as Rust links against libc. These three functions allow us to enter raw mode by setting and getting various terminal attributes:

pub fn get_attr() -> Termios {
          unsafe {
              let mut termios = core::mem::zeroed();
              tcgetattr(1, &mut termios);
              termios
          }
      }
      
      pub fn set_attr(termios: &mut Termios) {
          unsafe {
              tcsetattr(1, 0, termios);
          }
      }
      
      pub fn set_raw(termios: &mut Termios) {
          unsafe {
              cfmakeraw(termios);
          }
      }

Getting User Input

Now that we can read bytes individually, we can actually read the bytes. Let’s see how to do that.

use zui_core::term;
      use std::io;
      use zui_core::key::Key;
      
      fn main() {
          let mut stdout = io::stdout();
          let mut terminal = term::Terminal(&mut stdout).unwrap();
      
          for key in terminal.keys() {
              match key {
                  Key::Up => println!("Key went up"),
                  _ => break,
              }
          }
      }

This spawns a thread and passes bytes through a channel.

Blocks

You can create a block like this:

use std::io;
      use std::thread;
      use std::time::Duration;
      use zui_widgets::backend::ZuiBackend;
      use zui_widgets::layout::{Constraint, Direction, Layout};
      use zui_widgets::widgets::{Block, Borders, Widget};
      use zui_widgets::Terminal;
      use zui_core::key::Key;
      
      fn main() -> Result<(), io::Error> {
          let mut stdout = io::stdout();
          let backend = ZuiBackend::new(&mut stdout);
          let mut terminal = Terminal::new(backend)?;
      
          terminal.enter_raw_mode().unwrap();
          terminal.switch_screen().unwrap();
          terminal.draw(|f| {
              let size = f.size();
              let block = Block::default().title("Block").borders(Borders::ALL);
              f.render_widget(block, size);
          })?;
          for key in terminal.keys(io::stdin()) {
              terminal.set_cursor(2, 2).unwrap();
              match key {
                  Key::Ctrl('c') => break,
                  _ => println!("wrong"),
              }
          }
          Ok(())
      }

In this example, we demonstrate more functionality by creating a backend and a terminal. If we use zui_core alone, we wouldn’t need to do create a backend, but for widgets we do.

The terminal is basically the terminal from zui_core, just with methods to create widgets and position.

First we enter raw mode, and then we switch to another screen. Then, we draw a block and render it.

Finally, we iterate over the keys and print a message.
